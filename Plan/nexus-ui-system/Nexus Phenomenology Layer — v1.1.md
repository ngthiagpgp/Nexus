# Nexus Phenomenology Layer — v1.1

## Status

Versão para aprovação.

## Função deste documento

Este documento não existe para expandir teoria. Existe para travar, de forma operacional, como a tese do Nexus deve aparecer na interface.

A ontologia já foi definida.  
A doutrina espacial já foi definida.  
Os regimes cognitivos já foram definidos.

O problema agora não é mais conceitual.  
O problema agora é formal.

A pergunta que este documento responde é:

**como fazer o Nexus parecer um mundo operável, e não apenas uma interface bem organizada?**

---

# 1. Decisão central

O Nexus será desenhado como:

**uma superfície relacional de interpretação operacional**

E não como:

- dashboard sofisticado
    
- graph feature premium
    
- workspace de conhecimento
    
- conjunto de painéis analíticos
    

## Critério de aprovação

Uma tela do Nexus só pode ser considerada correta se, em menos de 5 segundos, ela fizer o usuário sentir que:

1. existe um centro claro de raciocínio;
    
2. existe um mundo relacional por trás dos elementos;
    
3. o detalhe está sob controle;
    
4. a interface serve julgamento, não apenas leitura.
    

---

# 2. Tradução operacional da ontologia

## Regra geral

Entidades diferentes devem parecer pertencer a naturezas diferentes.

Se ciclo, atividade, documento, evidência, risco, governança, decisão e trace parecem apenas variações do mesmo bloco, o Nexus falhou.

## Cycle

Deve parecer estrutura organizadora.  
Não pode parecer apenas cabeçalho administrativo.

## Activity

Deve parecer movimento operacional.  
Não pode parecer task card genérico.

## Document

Deve parecer objeto evidencial ou operacional situado.  
Não pode parecer arquivo neutro.

## Evidence

Deve parecer âncora epistêmica.  
Não pode parecer metadado disperso.

## Risk

Deve alterar leitura do sistema.  
Não pode parecer badge cromático.

## Governance

Deve comunicar legitimidade, limite ou responsabilidade.  
Não pode parecer burocracia decorativa.

## Decision

Deve parecer ponto de inflexão.  
Não pode parecer ação ordinária.

## Trace

Deve parecer rastro institucional interpretável.  
Não pode parecer log técnico com melhor estilo.

## Relation

Deve comunicar sentido.  
Não pode parecer linha genérica.

# 3. MAP Law — instrução de desenho

## Princípio

MAP não é visualização central.  
MAP é o ambiente primário de pensamento do Nexus.

## O que precisa aparecer visualmente

- soberania do centro
    
- sensação de campo contínuo
    
- possibilidade de foco e expansão sem perda de orientação
    
- relações com significado perceptível
    
- profundidade por intenção, não por excesso visível
    

## Regras duras

- o MAP não deve parecer encaixado como quadro principal da página;
    
- rails não podem competir semântica ou espacialmente com o centro;
    
- nodes devem ser leves, legíveis e não-cardificados;
    
- overlays devem revelar detalhe sem criar uma segunda arquitetura;
    
- relações precisam variar quando o significado variar.
    

## Teste de validação

Se esconder os rails por alguns segundos e a tela perder seu sentido, o MAP ainda não é soberano.

## Sinal de fracasso

O usuário sente que está vendo um grafo importante.

## Sinal de sucesso

O usuário sente que está pensando dentro do campo relacional do sistema.

---

# 4. Inspect Law — instrução de desenho

## Princípio

Inspect não é tela de detalhes.  
Inspect é regime de julgamento.

## O que precisa aparecer visualmente

- o objeto ativo domina semanticamente a leitura;
    
- conteúdo, relação e evidência parecem partes do mesmo ato cognitivo;
    
- o suporte fica disponível sem fragmentar a experiência;
    
- a tela parece superfície contínua, não composição de blocos.
    

## Regras duras

- se Inspect parecer três colunas organizadas, falhou;
    
- se a área de leitura for pequena, falhou;
    
- se o metadado aparecer mais forte que o conteúdo, falhou;
    
- se o usuário sentir que abriu um “painel de detalhe”, falhou.
    

## Teste de validação

O usuário deve sentir pressão para formar juízo antes de terminar de explorar toda a tela.

## Sinal de fracasso

O usuário entende mais, mas não sente necessidade de decidir.

## Sinal de sucesso

O usuário sente que entrou em estado de análise e julgamento.

---

# 5. Audit Law — instrução de desenho

## Princípio

Audit não é cronologia premium.  
Audit é memória institucional interpretável.

## O que precisa aparecer visualmente

- agrupamento semântico;
    
- transições significativas;
    
- cadeia causal;
    
- rastro institucional;
    
- redução de repetição perceptiva.
    

## Regras duras

- se parecer feed, falhou;
    
- se parecer timeline sofisticada, falhou;
    
- se parecer log técnico refinado, falhou;
    
- se o usuário souber o que aconteceu, mas não por que importou, falhou.
    

## Teste de validação

O usuário deve conseguir inferir a lógica da mudança sem precisar ler item por item.

## Sinal de fracasso

A interface informa sequência, mas não reconstrói sentido.

## Sinal de sucesso

A interface transforma histórico em estrutura de memória e consequência.

# 6. Política de forma

## O que aumentar

- soberania do centro
    
- diferença entre naturezas ontológicas
    
- legibilidade das relações
    
- profundidade sob demanda
    
- contraste entre regimes
    

## O que reduzir

- bounding perceptual excessivo
    
- modularidade visível demais
    
- metadado persistente
    
- simetria que enfraquece o centro
    
- densidade distribuída democraticamente
    

## O que proibir

- cardificação do MAP
    
- Inspect como layout SaaS de detalhes
    
- Audit como feed/timeline estilizada
    
- rails semânticamente ambiciosos demais
    
- profundidade exibida cedo demais
    

---

# 7. Missão do próximo sprint de design

O próximo sprint não deve expandir teoria.  
O próximo sprint deve provar visualmente três leis:

1. **MAP como mundo inevitável**
    
2. **Inspect como julgamento inevitável**
    
3. **Audit como prova estrutural**
    

Cada frame deve ser avaliado contra uma única pergunta:

**isto revela um mundo operável — ou apenas uma interface bem organizada?**

---

# 8. Regra final de aprovação

Esta camada está aprovada quando cumprir duas funções ao mesmo tempo:

1. impedir deriva conceitual adicional;
    
2. orientar desenho de interface com critérios suficientemente duros para crítica.
    

Se ela ainda gerar debate abstrato demais, falhou.  
Se ela já permitir dizer com clareza por que um frame está certo ou errado, cumpriu sua função.
