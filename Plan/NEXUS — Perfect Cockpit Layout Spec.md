## v1.0 (Final Institutional Operating Surface)

---

# 1. Princípio absoluto do layout

O cockpit deve permitir:

→ entender situação institucional em **3 segundos**.

Não em:

- 30 segundos
    
- 3 minutos
    
- depois de navegar
    

A percepção inicial deve responder:

- onde estou institucionalmente
    
- o que exige ação
    
- onde está risco
    
- qual evidência importa
    

Layout é:

**modelo mental espacializado**

---

# 2. Estrutura macro do cockpit

Layout final:

┌─────────────────────────────────────────────────────┐  
│ GLOBAL CONTEXT BAR                                  │  
├───────────────┬───────────────────────┬─────────────┤  
│ ONTOLOGY      │ SYSTEM REALITY        │ DECISION     │  
│ NAVIGATION    │ (GRAPH CORE)          │ & EVIDENCE   │  
├───────────────┴───────────────────────┴─────────────┤  
│ EXECUTION FLOW                                     │  
└─────────────────────────────────────────────────────┘

Quatro zonas:

1. Contexto institucional global
    
2. Universo institucional navegável
    
3. Realidade sistêmica viva (grafo)
    
4. Tribunal decisório
    
5. Fluxo operacional
    

---

# 3. Global Context Bar

Função:

→ situar operador no sistema institucional.

Deve mostrar:

- ciclo ativo principal
    
- ciclos críticos secundários
    
- pressão temporal
    
- risco agregado
    
- estado epistêmico global
    

Formato:

- barra horizontal fixa
    
- leitura em uma linha
    
- sem clutter
    

Não pode conter:

- botões administrativos
    
- metadata técnica
    
- status irrelevante
    

É:

**painel de consciência institucional.**

---

# 4. Ontology Navigation Panel (Left)

Não é menu.

É:

**mapa institucional estrutural.**

Contém:

- ciclos
    
- entidades
    
- sistemas
    
- autoridades
    
- mandatos
    

Formato:

- árvore híbrida + mini grafo
    
- foco estrutural
    
- seleção altera grafo central
    

Regras:

- não mostrar atividades aqui
    
- não mostrar documentos aqui
    
- não mostrar logs aqui
    

Isso é:

→ estrutura, não operação.

---

# 5. System Reality Core (Center)

Este é o coração do Nexus.

Sempre:

**grafo-first**

Não lista.  
Não cards.  
Não dashboard.

Grafo ocupa:

→ ~60% da tela.

Funções:

- perceber tensão institucional
    
- perceber bloqueios
    
- perceber dependências
    
- perceber risco emergente
    

Comportamentos:

- zoom contextual
    
- cluster dinâmico
    
- highlight causal
    
- overlay temporal
    

Estados visuais:

- normal
    
- tensão
    
- crise
    
- resolução
    

Este centro deve parecer:

→ sistema vivo.

---

# 6. Decision & Evidence Panel (Right)

Não é detail panel.

É:

**zona de julgamento institucional.**

Estrutura interna:

### Primary Object Focus

Objeto central atual.

### Evidence Stack

Base epistêmica.

### Governance Chain

Cadeia de legitimidade.

### Risk Envelope

Impacto sistêmico.

### Agent Interpretation

Leitura analítica sugerida.

Este painel deve permitir:

- decidir
    
- justificar
    
- escalar
    

Não apenas:

- ler dados.
    

---

# 7. Execution Flow Layer (Bottom)

Função:

→ realidade operacional temporal.

Componentes:

### Institutional Kanban

- atividades relevantes
    
- bloqueios
    
- dependências
    

### Timeline Ribbon

- evolução temporal
    
- deadlines críticos
    

### Decision Pipeline

- decisões pendentes
    
- decisões tomadas
    
- decisões contestadas
    

Este layer responde:

→ o que fazer agora.

---

# 8. Navegação primária final

Modo global:

MAP | FLOW | INSPECT | AUDIT

Mas:

- MAP = default
    
- FLOW = execução
    
- INSPECT = profundidade
    
- AUDIT = narrativa
    

Mudança de modo não muda layout.

Muda:

→ projeção perceptiva.

---

# 9. Hierarquia cognitiva

Layout define ordem de leitura:

1. Global Context → situação
    
2. Graph Core → sistema
    
3. Decision Panel → julgamento
    
4. Flow Layer → ação
    
5. Ontology → estrutura
    

Se operador começa pela esquerda:

layout falhou.

Centro é primeiro.

---

# 10. Visual density rules

Centro:

- alta densidade informacional
    
- baixa densidade textual
    

Direita:

- média densidade textual
    
- alta clareza semântica
    

Baixo:

- média densidade operacional
    

Esquerda:

- baixa densidade
    
- estrutura pura
    

Topo:

- ultra sintético
    

---

# 11. Temporal integration

Tempo aparece:

- no topo → pressão global
    
- no grafo → evolução estrutural
    
- no flow → execução
    
- no audit → narrativa
    

Tempo nunca é:

→ label escondida.

---

# 12. Multi-cycle awareness

Cockpit deve permitir:

- ver múltiplos ciclos simultaneamente
    
- perceber conflito
    
- perceber dependência
    

Modo crise:

→ grafo reorganiza prioridade visual.

---

# 13. Agent presence

Agente não é chat lateral.

É:

- camada cognitiva embutida
    
- highlight interpretativo
    
- insight contextual
    
- projeção de risco
    

Agente deve parecer:

→ inteligência sistêmica.

---

# 14. Estados perceptivos do cockpit

Cockpit pode estar em:

### Calm

Operação normal.

### Tension

Risco emergente.

### Crisis

Colapso institucional iminente.

### Resolution

Reorganização em curso.

UI muda sutilmente:

- cor
    
- animação
    
- cluster
    

---

# 15. Performance perceptiva alvo

Operador deve:

- entender ciclo em <3s
    
- localizar bloqueio em <5s
    
- identificar evidência crítica em <8s
    
- justificar decisão em <20s
    

Se não:

layout falhou.

---

# 16. Anti-patterns proibidos

- dashboard com widgets
    
- painel lateral com scroll infinito
    
- grafo pequeno
    
- múltiplos contextos simultâneos
    
- modo “admin panel”
    
- workflow tool clássico
    
- estética devtool
    
- navegação CRUD
    

---

# 17. Critério final de perfeição

Cockpit perfeito quando:

- parece sala de situação institucional
    
- parece sistema vivo
    
- complexidade é navegável
    
- decisão parece inevitável
    
- auditoria parece natural
    
- operador sente controle sistêmico
