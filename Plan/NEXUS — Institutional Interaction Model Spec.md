## v1.0 (Foundational)

---

## 1. Natureza da interação

O Nexus não é:

- sistema de tarefas
    
- sistema documental
    
- BI
    
- workflow engine
    

Ele é:

**campo operacional institucional navegável**

Usuário não executa ações isoladas.  
Usuário atua dentro de:

- mandatos temporais
    
- restrições evidenciais
    
- dependências sistêmicas
    
- accountability institucional
    

Logo:

Interação ≠ clique  
Interação = **ato institucional**

---

## 2. Unidade fundamental de ação

### Cycle

Cycle é:

Mandato operacional delimitado no tempo  
+ com escopo institucional explícito  
+ com base evidencial necessária  
+ com cadeia de responsabilidade rastreável

Toda interação ocorre dentro de um cycle.

Não existe:

- ação fora de ciclo
    
- documento fora de ciclo
    
- decisão fora de ciclo
    

Se existir → erro ontológico.

---

## 3. Papéis institucionais no sistema

### Operator

Executa trabalho operacional.

### Reviewer

Valida legitimidade e coerência.

### Authority

Autoriza decisões institucionais.

### Agent

Interpreta sistema e sugere ação.

### System

Garante integridade estrutural.

Interações mudam conforme papel.

UI deve refletir isso implicitamente.

---

## 4. Estrutura de realidade institucional

Cada ciclo possui 5 camadas simultâneas:

### Operacional

Atividades concretas.

### Epistêmica

Documentos, evidência, conhecimento.

### Governança

Autorização, responsabilidade, decisão.

### Temporal

Janela institucional e deadlines.

### Sistêmica

Dependências com outros ciclos.

Sistema falha se qualquer camada invisível.

---

## 5. Fluxo institucional de decisão

Modelo básico:

Situação detectada  
→ atividade criada  
→ evidência requerida  
→ evidência avaliada  
→ decisão tomada  
→ ação executada  
→ resultado registrado  
→ memória institucional formada

Nexus deve tornar isso perceptível.

Hoje está implícito.

Precisa ser explicitamente navegável.

---

## 6. Natureza das atividades

Atividade não é task.

Atividade é:

**ato operacional institucionalmente relevante**

Ela pode:

- gerar risco sistêmico
    
- exigir validação epistêmica
    
- demandar autoridade
    
- impactar múltiplos ciclos
    

Logo:

atividade tem peso ontológico.

UI precisa mostrar:

→ gravidade institucional da atividade.

---

## 7. Natureza dos documentos

Documento não é arquivo.

Documento é:

**unidade de legitimidade operacional**

Ele pode:

- desbloquear ação
    
- invalidar decisão
    
- alterar prioridade
    
- redefinir ciclo
    

Estados epistemológicos:

- trusted
    
- draft
    
- drift
    
- contested
    
- obsolete
    

UI precisa tornar isso visceral.

---

## 8. Natureza do risco

Risco não é alerta técnico.

Risco é:

**potencial de falha institucional**

Tipos:

- evidential risk
    
- temporal risk
    
- governance risk
    
- systemic risk
    

Interação correta:

Operador deve perceber risco antes de ler logs.

---

## 9. Governança operacional

Toda decisão deve responder:

- quem decidiu
    
- com base em quê
    
- quando
    
- qual impacto sistêmico
    

Isso não é relatório posterior.

Isso é parte da interação.

---

## 10. Temporalidade institucional

Tempo não é metadata.

Tempo é:

**estrutura organizadora da realidade operacional**

Cycle = tempo institucional.

UI deve mostrar:

- urgência real
    
- atraso sistêmico
    
- sobreposição de mandatos
    
- janela crítica
    

---

## 11. Memória institucional

Audit não é histórico técnico.

É:

**narrativa institucional**

Memória precisa permitir:

- reconstruir decisões
    
- explicar falhas
    
- legitimar ações futuras
    

---

## 12. Interação humano-agente

Agente não executa.

Agente interpreta.

Ele:

- detecta incoerência
    
- propõe reorganização
    
- identifica lacunas evidenciais
    
- antecipa risco
    

Humano decide.

Sistema registra.

---

## 13. Interação ideal (sequência perceptiva)

Operador entra no Nexus e imediatamente percebe:

1. qual ciclo define sua realidade
    
2. qual atividade exige atenção
    
3. qual evidência está faltando
    
4. qual decisão está bloqueada
    
5. qual risco está crescendo
    
6. qual ação institucional é necessária
    

Se qualquer um destes exige navegação profunda → falha.

---

## 14. Critério de sucesso do modelo

Sistema funciona quando:

- operador pensa melhor com o Nexus do que sem ele
    
- decisões ficam mais justificáveis
    
- falhas ficam previsíveis
    
- auditoria se torna natural
    
- organização ganha memória real
    

---

## 15. Consequência estrutural para UI

A UI deve ser:

- grafo-first
    
- decisão-first
    
- evidência-first
    
- ciclo-centric
    
- risco-visceral
    
- governança-nativa
    

Não pode ser:

- lista-first
    
- CRUD-first
    
- documento-first
    
- tarefa-first
