## v1.0 (Operational Layer)

---

# 1. Princípio mecânico central

Toda interação no Nexus deve responder:

What institutional state changes because of this action?

Se a resposta for:

→ “nenhuma”  
então a interação é ruído.

---

# 2. Estados mecânicos do operador

O operador sempre está em um de 4 modos cognitivos:

### ORIENT

Entender situação institucional.

### DIAGNOSE

Identificar bloqueios, riscos, incoerências.

### DECIDE

Escolher curso institucional de ação.

### EXECUTE

Registrar ato operacional.

UI deve permitir transição fluida entre estes modos.

Hoje está misturado.

---

# 3. Navegação mecânica primária

Navegação principal:

MAP → FLOW → INSPECT → GOVERN

Não é navegação visual.  
É navegação cognitiva.

---

# 4. MAP mechanics

MAP é o estado default.

Objetivo:

→ formar modelo mental sistêmico.

Interações:

### Click node

→ torna nó centro cognitivo.

### Drag

→ explorar estrutura relacional.

### Hover edge

→ revela natureza institucional da dependência.

### Multi-select

→ análise sistêmica (ex: risco agregado).

### Zoom temporal

→ ver evolução do ciclo.

MAP nunca é:

- decoração
    
- overview estático
    
- feature opcional
    

---

# 5. FLOW mechanics

FLOW é:

→ realidade operacional linear.

Submodes:

### Activity Flow

Kanban institucional.

### Evidence Flow

Fluxo epistemológico.

### Timeline

Sequência institucional temporal.

Interações:

- mover atividade muda estado institucional
    
- anexar documento altera legitimidade
    
- marcar bloqueio altera topologia de risco
    

---

# 6. INSPECT mechanics

INSPECT não é detail view.

É:

→ tribunal institucional.

Estrutura:

### Context tab

Onde este objeto vive no sistema.

### Evidence tab

Qual base epistêmica sustenta.

### Governance tab

Quem responde institucionalmente.

Interações:

- avaliar evidência
    
- reconciliar drift
    
- elevar risco
    
- solicitar autoridade
    

---

# 7. GOVERN mechanics

GOVERN é camada explícita.

Objetivo:

→ agir sobre legitimidade institucional.

Ações possíveis:

- approve
    
- escalate
    
- freeze
    
- override
    
- mandate
    
- archive
    

Cada ação deve:

- gerar evento institucional
    
- alterar grafo
    
- alterar fluxo
    
- alterar memória
    

---

# 8. Document mechanics

Documento possui estados ativos.

Operações:

### attach

liga documento a atividade.

### validate

muda peso epistemológico.

### drift

detecta inconsistência.

### reconcile

restaura confiança.

### supersede

substitui base evidencial.

Cada operação altera:

→ estrutura decisória.

---

# 9. Risk mechanics

Risco não é badge.

Risco altera:

- prioridade
    
- visibilidade
    
- topologia do grafo
    
- ordem do fluxo
    

Tipos mecânicos:

- local risk
    
- systemic risk
    
- legitimacy risk
    
- temporal risk
    

UI deve tornar risco:

→ perceptivamente dominante.

---

# 10. Temporal mechanics

Tempo atua como força.

Operações:

- deadline shift altera pressão institucional
    
- ciclo overlap altera complexidade sistêmica
    
- atraso gera risco emergente
    

MAP deve permitir:

→ visualização temporal contínua.

---

# 11. Agent interaction mechanics

Agente opera em camada interpretativa.

Operações:

### summarize_system_state

gera leitura institucional.

### detect_incoherence

marca tensão estrutural.

### suggest_restructuring

propõe reorganização de ciclo.

### simulate_outcome

antecipa impacto.

Agente nunca executa.

---

# 12. Multi-cycle mechanics

Sistema permite:

- dependência entre ciclos
    
- hierarquia institucional
    
- conflito de mandatos
    

MAP deve suportar:

→ visão multi-ciclo simultânea.

---

# 13. Memory mechanics

Cada ação gera:

institutional event object.

Memória permite:

- replay institucional
    
- auditoria narrativa
    
- análise causal
    

---

# 14. UI feedback mechanics

Toda interação deve produzir:

1. mudança perceptiva no sistema
    
2. alteração estrutural visível
    
3. consequência institucional clara
    

Sem isso:

interação inválida.

---

# 15. Anti-mechanics (proibido)

- abrir formulário sem impacto sistêmico
    
- editar campo sem alterar realidade
    
- criar task sem contexto institucional
    
- anexar documento sem efeito epistêmico
    
- navegar lista sem modelo mental
    

---

# 16. Sequência mecânica ideal

Operador entra:

1. MAP mostra tensão institucional
    
2. FLOW mostra trabalho crítico
    
3. INSPECT mostra evidência faltante
    
4. GOVERN permite decisão
    
5. memória registra evento
    
6. sistema reorganiza
    

Isso é loop fundamental.

---

# 17. Critério de implementação correta

Mecânica está correta quando:

- operador pensa em termos institucionais
    
- sistema altera comportamento humano
    
- decisões ficam estruturadas
    
- risco torna-se previsível
    
- organização ganha inteligência
